Key4ce

Beta Cross-platform

Terminal typing game with real keystroke analytics. No fluff, no internet, no accounts.

What it is

Key4ce is a terminal-based typing speed test that actually tells you something useful about how you type. Beyond WPM and accuracy, it tracks per-key latency, bigram hesitation patterns, and error frequency by finger. The idea is that knowing you average 94 WPM is less useful than knowing you consistently fumble the right-hand top row and pause 40ms longer on every 'n' than any other letter.

It runs entirely in the terminal, stores nothing remotely, and works on Windows, macOS, and Linux with a single binary. No Electron, no web view — actual curses-style terminal UI.

How it works

key4ce — session #47
key4ce start --mode 1min --wordset common-1k
─────────────────────────────────────────────
the quick brown foxe jumps over the lazy dog and kept running
─────────────────────────────────────────────
WPM 87
Accuracy 96.4%
Elapsed 00:23
Errors 1
Slowest key n (+38ms)

Keystrokes are captured using raw terminal input mode — the app puts the terminal in cbreak mode to get each keypress immediately without waiting for Enter. Every keypress is timestamped in microseconds. After the session, that timestamp array is walked to compute inter-key intervals, bigram pairs, and error positions. The analytics report is rendered right back in the terminal at the end of the session.

Raw keypress → Timestamp → Compare to target → Update live stats → End → report

How it was built

Python with the curses standard library for the terminal UI and windows-curses for Windows compatibility. Packaged to a single binary using PyInstaller — no Python installation required on the target machine. Session history is stored locally in a SQLite file in the user's home directory.

Cross-platform raw input handling was the main pain point. curses on Windows behaves differently from POSIX curses in ways that aren't well-documented. The Windows path required a separate input loop and careful handling of arrow keys and function keys that Windows terminal emulators represent differently.

What I learned

Microsecond-precision timing in Python is less reliable than you'd expect because of the GIL and garbage collection pauses. The latency numbers are real but noisy — I had to apply a light median filter over rolling windows to make the per-key stats stable enough to be meaningful. That was a good lesson in the difference between measuring something precisely and measuring it accurately.

The other thing I underestimated was word set design. Common English word lists are dominated by short function words, which produce unrealistically high WPM numbers. I ended up curating a few different word sets — common-1k, programming-keywords, punctuation-heavy — and making them selectable. Users who care about accurate measurement want the harder sets.
