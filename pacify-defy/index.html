Pacify & Defy AI

In Dev Full-Stack

One prompt. Two personalities. Balanced or unhinged — you pick.

What it is

Pacify & Defy AI is a dual-mode conversational AI web app. Pacify mode is measured, direct, well-reasoned — the kind of response you'd want when you actually need help thinking through something. Defy mode has the guardrails removed: blunter, weirder, willing to go places the other mode won't. Same model underneath, very different system prompt and temperature. The idea is to make the personality switch explicit and user-controlled rather than hidden behind vague "tone" settings.

Each message triggers a streaming completion request to the API. Pacify uses a conservative temperature and a system prompt focused on accuracy and nuance. Defy uses a higher temperature, a very different system prompt, and no hedging instructions. The mode switch is instantaneous — users can flip mid-conversation, though the history carries over so the contrast becomes even more apparent.

User message → Mode select → System prompt → Stream tokens → Render live

How it was built

Next.js frontend, Supabase for auth and data. The streaming is done properly — the API route uses TransformStream to pipe tokens from the model API to the client as they arrive, so the response renders character by character rather than appearing all at once after a delay.

Auth is real — JWT-based sessions, row-level security in Supabase so users can only see their own conversation history. Each user's Pacify and Defy conversation histories are stored completely separately; mixing them felt wrong UX-wise and creates cleaner data boundaries. The API key never touches the client.

What I learned

Streaming responses are significantly more complex to implement correctly than I expected. Getting the token stream to render smoothly without layout thrashing required thinking carefully about how React's reconciler handles rapid state updates. Batching the token appends and using a ref for the in-flight message before committing to state made a noticeable difference.

The other thing I didn't anticipate was how much of the project ended up being about the system prompts rather than the code. The difference between a good Defy response and a bad one is entirely in the prompt. I spent more time iterating on those than on any individual feature.
